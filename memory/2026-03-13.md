  - Data URL used for photo preview (converts file to base64)
- **Status Toggle**: Separate button (啟用/停用) instead of combining with delete
- **Soft Delete**: DELETE API call marks as inactive rather than hard delete

  - Step 1: Clinic → Step 2: Doctor → Step 2b: Service → Step 3: Date → ...
  - Progress bar updated to show "2b" as separate step
  - Back button from Step 3 goes to 2b, not 2
- **Doctor Cards**:
  - 80x80px circular photo or default emoji avatar (👨‍⚕️)
  - Type badge uses accent-green background
  - Services listed as bullet points below description
  - Shows price if available
- **Service Filtering**: 
  - Services now loaded from selected doctor's services array
  - Falls back to filtering by doctorId if services array missing
- **Data Flow**:
  - bookingData.doctorId, doctorName, doctorType stored after selection
  - bookingData.servicePrice added for future payment integration
  - Payload includes doctorName and serviceName for display

#### API Endpoints Used:
- Admin: `/api/admin/doctors` (GET/POST/PUT/DELETE)
- Public: `/api/doctors` (GET)

#### CSS Classes Reused:
- `.option-card` for doctor selection (same as clinic/service cards)
- `.status-badge` with `.status-confirmed`/`.status-cancelled` for active/inactive
- `.modal-overlay`, `.modal`, `.modal-header/body/footer` for doctor modal
- `.service-row` custom class for dynamic service inputs

#### Edge Cases Handled:
- Empty doctor list → shows fallback default doctors
- No services for doctor → shows error message, disables next button
- Inactive doctors → filtered out on booking page, shown as "停用" on admin
- Photo upload failure → falls back to default avatar
